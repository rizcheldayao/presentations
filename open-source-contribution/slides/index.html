      Case study

      IBM Cloud and Carbon Components

      What is IBM Cloud?

    • Enterprise software
    • No open source whatsoever
    • Broken down into microservices

      • picture of IBM Cloud here

      What is Carbon components?

    • Component library and design system
    • Vanilla JS, React components and style guidelines
    • All open source

      • picture of Carbon Components

      How did they differ?

      IBM Cloud:

      • One person controlled all the code
      • No peer reviews to ensure security and accountability
      • Quality of code was bad

        • Carbon Components

        • Contributions from anyone
        • Peer reviews on all contributions
        • Testing
        • Good quality of code
        • Open for anyone to use

          • IBM Cloud code has improved because of the influence of Carbon Components üíØ

          It now has...

        • A peer review system
        • At least 85% testing coverage
        • Pair programming
        • Knowledge of the system among the whole team

                      It can be scary when first contributing

                      There are many fears we face...

                      • I'm just a new developer
                      • Criticism for my submission
                      • What if my contribution is not accepted?

                        • Here are some ways to overcome those fears...

                        I'm just a new developer

                        • You don't need to code to contribute back
                        • Everyone started somewhere
                        • Ask help from a more senior developer

                          • Criticism for my submission

                          • Ask for peer review
                          • Improve your skills by creating pull requests
                          • You will _usually_ get constructive feedback
                          • It will make you a better developer

                            • What if my contribution is not accepted?

                            • This could be because of several things
                            • Overwhelmed maintainers or resolved issues
                            • Look for repos with active maintainers that merge PRs quite often
                            • The best thing is to move on and contribute to something

                              • General common pitfalls:

                              • Establishing coding standards
                              • Peer reviews
                              • Maintenance
                              • Documentation

                                • Contributor common pitfalls:

                                • Open source licensing
                                • Contributing for the sake of contributing
                                • Nothing is perfect
                                • Document and comment your code

                                  • 1. Search for specific labels

                                  2. Find easy fixes

                                  3. Push up your changes in a PR
