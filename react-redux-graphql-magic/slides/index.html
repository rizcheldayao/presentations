    Building enterprise applications is hard

    It can make you feel like a queen (or king) when you're on a roll

    Or it can make you feel like throwing your computer if things aren't going well

    One of the most challenging tasks is designing the architecture and set up

    There are so many things we have to consider:

    1. What developer tools (language, library, etc) best meet our needs?

    2. How do we manage our internal and external data within our microservice?

    3. How do we maintain clean and consistent code?

    4. How do we create a consistent way to configure and deploy all our microservices?

    Our team was tasked to create a new front end microservice for our virtual server ordering experience in IBM Cloud

    Like any project, we had certain requirements to consider:

  • Was this even possible to integrate Softlayer APIs into the IBM Cloud platform?
  • New microservice that fit into the existing ecosystem
  • Consume both IBM Cloud & Softlayer APIs
  • Testing & component based architecture

    • Challenge #1: What developer tools best meet our needs?

    React, ES6, Express, Webpack

    Why did we choose React?

  • Reusable components
  • Fast render with Virtual DOM
  • Redux state management
  • Data binding
  • Carbon components library

    • React is a JavaScript library for building user interfaces

  • Functions as the view layer
  • Component based
  • Virtual DOM
  • Declarative
  • JSX

    React containers

    Stove

    Made up by smaller parts

    Manages everything within its ecosystem

    Passes the state to smaller parts

    Containers

    Made up of smaller elements called components

    Manages all of the data

    Passes data as props to components

    React components

    Stove burner

    Reusable piece

    Receive data and state from stove

    Components

    Reusable in multiple parts of an application

    Receives data and state from container

    Accept data from containers through props

    A separation of concerns allows an application to be easier to manage and provides cleaner code

  • Actions describe something has happened
  • Reducers update the state
  • Store holds the application state
