    Challenge #2: How do we manage our internal and external data within our microservice?

    Redux and GraphQL

    GraphQL was life changing üôåüèº

    GraphQL is a data query language for your APIs:

  • Send GraphQL query and get exactly what you need
  • Gets many resources in a single request
  • Apollo client cache query results
  • Support local, offline development

    • https://www.apollographql.com/client/

    We created an internal package to allow independent data sources to be composed into a single GraphQL schema that was later open sourced...GrAMPS

    Data source breakdown:

  • Models: functions for the different calls made with the parameters
  • Resolvers: contain mutations and queries
  • Schema: schema of the api calls (parameters, returned data, types)
  • Mocks: mock local data using casual

    • This allowed:

  • A central GraphQL microservice that anyone could consume
  • An isolated data source for each microservice
  • Easy adoption
  • Better performance

    • To authenticate, we needed to create a middleware layer

    Middleware consist of a series of function calls:

  • Access request object, response object, and next middleware function
  • Able to make changes to request and response objects
  • Chain multiple functions

    • function logger(req,res,next){

    console.log(new Date(), req.method, req.url);

    next();

    }

    Middleware flow:

    1. Initiate authentication from the front end

    2. Specified routes in middleware

    3. Perform token exchange and send back auth object to front end

    4. Pass user id and token as props to containers

    Initiate authentication from front end:

    Required user id and token in our React containers to make API calls

    Made an XMLHttpRequest in componentWillMount lifecycle method

    Specified the account id and URL to call in our middleware layer

    The request only had a url and method

    Specified routes in middleware:

    Used Express Router

    router.get('/api/SLAuth', getSLID, getSLToken);

    Perform token exchange and send back to front end:

    Searched accounts to get user id then sent the user id in the request object to the next function

    Use Request library to make POST call for token

    Send user id and token back as response

    Response object was passed as props down to containers

    Challenge #3: How do we maintain clean and consistent code?

    Testing and pull request reviews

    Testing will make your life easier in the long run

    Jest and Enzyme are unit testing utilities to test a React application:

  • Shallow rendering
  • Full DOM rendering
  • Coverage reports
  • Snapshot testing
  • Functionality testing

    • Shallow rendering: Shallow rendering is useful to constrain yourself to testing a component as a unit, and to ensure that your tests aren't indirectly asserting on behavior of child components.

    Full DOM rendering (mount): Full DOM rendering is ideal for use cases where you have components that may interact with DOM APIs, or may require the full lifecycle in order to fully test the component. Full DOM rendering requires that a full DOM API be available at the global scope. This means that it must be run in an environment that at least ‚Äúlooks like‚Äù a browser environment.

    Snapshot: A typical snapshot test case for a mobile app renders a UI component, takes a screenshot, then compares it to a reference image stored alongside the test.

    Testing:

    With mock data

    By simulating events

    For state or prop change

    Test actions and reducers to equal the correct action and state

    Require at least 75% testing coverage

    Pull requests show what someone has changed to push up to a GitHub repository:

  • Require at least 2 reviews
  • Pass all tests in the CI/CD pipeline
  • Ensures all code is reviewed by someone else

    • Challenge #4: How do we create a consistent way to configure and deploy all our microservices?

    Kubernetes

    Kubernetes is an orchestration tool:

  • Automates deployment
  • Manages the deployment & containerized applications

    • Kubernetes is like the home builder in charge of scheduling and managing a house

    Kubernetes schedules pods go to which nodes, manage deployment

    Kubernetes definitions:

  • Cluster: Consist of a cluster master and multiple worker machines
  • Node: Worker machine that run applications managed by the cluster master
  • Pod: Usually contains one container

    • House equivalents:

    = Home builder company

    = Home builder employees

    = Room

    Kubernetes at an enterprise scale:

    1. What happens if my app crashes?

    2. How can I roll out updates or rollbacks?

    3. What happens when I get an unexpected high influx of traffic?

    4. What will work best for my microservice architecture?

    What happens if my app crashes?

    Provides monitoring for pod health and restarts them in case of failure

    How can I roll out updates or rollbacks?

    A user can roll out or rollback an update with zero downtime by incrementally updating Pods instances with new ones

    What happens when I get an unexpected high influx of traffic?

    Automatically scales the number of pods in deployment

    What will work best for my microservice architecture?

    Microservices can be packaged into containers which are each deployed as pods for isolation and resource sharing
